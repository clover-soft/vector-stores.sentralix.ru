### Шаг 10: Доменные ручки управления загрузками файлов в провайдера (не admin)
- Цель:
  - Дать фронтенду возможность управлять загрузками локальных файлов в провайдера без доступа к админ-разделу.
  - Обеспечить получение `external_file_id`, статуса и диагностических ошибок по загрузке.
- Действия:
  - Добавить ручку “получить/создать upload” для файла:
    - `POST /api/v1/files/{file_id}/provider-uploads/{provider_type}`
    - Вход:
      - `file_id: str`
      - `provider_type: str`
      - `X-Domain-Id: str`
      - (опционально) `force: bool` (query)
      - (опционально) `meta: dict` (body)
    - Выход:
      - объект upload (аналог `ProviderFileUploadOut`).
    - Поведение:
      - Проверить, что локальный файл принадлежит домену.
      - Создать/обновить `rag_provider_file_uploads` через `get_or_sync`.
      - Важно (для отложенной индексации):
        - эта операция отвечает только за наличие файла у провайдера (Files API) и не должна ждать индексирование в vector store.
        - результат (`external_file_id`) используется далее при прикреплении файла к индексу у провайдера (attach) и при последующем polling статусов.
        - для `provider_type="yandex"` `external_file_id` совпадает с `vector_store.file.id`.
  - Добавить ручку списка upload-ов по файлу:
    - `GET /api/v1/files/{file_id}/provider-uploads`
    - Вход: `X-Domain-Id`
    - Выход: список upload-ов по всем провайдерам (или по фильтру `provider_type`).
- Ожидаемый результат:
  - Фронтенд может подготовить файл к использованию в индексе у выбранного провайдера и видеть статус загрузки.
- Критерии готовности (Definition of Done):
  - Реализованы ручки `POST /api/v1/files/{file_id}/provider-uploads/{provider_type}` и `GET /api/v1/files/{file_id}/provider-uploads`.
  - Доменная изоляция соблюдена.
  - Корректно обрабатываются ошибки провайдера (400/502) и ошибки отсутствия файла (404).
  - Swagger документирован.
- Выполнено: нет
